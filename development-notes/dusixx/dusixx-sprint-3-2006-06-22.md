# Спринт 3: Директивы, пайпы, формы — 2026-06-22

## Что было сделано:

### 1. Jamendo API 

Реализовал базовый Jamendo API сервис на базе HTTP client. Создал провайдер с основными параметрами конфигурации API. Добавил интерсептор для обработки `5xx` ошибок и `ошибок сети.

### 2. Cache

Реализовал простой `постраничный` кеш, для сохранения результатов запросов. Данные для страниц могут дублироваться, но логика простая и предсказуемая. Так же реализовал кеш, в котором снапшоты страниц хранят не сами данные, а id своих сущностей. 

Сами сущности хранятся отдельно и не дублируются. Чтобы избежать конфликтов между страницами и их данными - добавил счетчики для сущностей. При удалении страниц счетчики уменьшаются на единицу. При добавлении страницы в кеш у всех ее сущностей, уже находящихся в нем, счетчик увеличивается на единицу. Пока значение счетчика больше нуля - сущность остается в кеше. 

Так же добавил время жизни (`ttl`) для страниц. По истечении кторого, при попытке получить страницу из кеша она удаляется, вынуждая сделать запрос за новыми данными. Ключи для кеша создаются из параметров запроса (`key=value`), отсортированных по `key` в алфавитном порядке с добавлением специфичного префикса. Учитывая лимит на кол-во запросов, даже самый простой кеш вполне вариант.

### 3. TrackService

Создал кешируемый сервис для треков, с обработкой ошибок API и `маппингом` данных. Релизовал абстрактный `store service` для результатов API запросов с функционалом `пагинации` и состояниями `load` и `error`. Сервис наследуется и может расширяться (например, для поиска). Решил пока оставаться в рамках сервисов, а `nrgx` использовать для глобального состояния.

## Проблемы и решения:

Оказалось, что при поиске каждый следующий запрос (с измененным `offset`) может возвращать дубли. Ангуляр закономерно ругается при рендеринге такого списка. Грешил на проблемы в коде - оказалось так работает API. Сделал дедупликацию результатов.
