# Sprint 3 Team Discussion: UI, Architecture & Workflow Decisions

**Date: 2026-06-17**

## What was discussed

**`Playlist` creation flow:**
- discussed creating a separate route for `Playlist` page
- discussed advantages of using a dedicated route instead of placing playlist creation inside `Library` page
- agreed that a separate route improves navigation, browser history support, and future scalability

**Horizontal card component:**
- discussed the design of a horizontal card for playlists, tracks, and search results
- considered creating a universal reusable component
- reviewed how this approach can reduce code duplication

**`Library` and `Playlist` page design:**
- analyzed `Library` and `Playlist` pages in Spotify and Deezer
- discussed which service provides a more suitable user experience for the project
- since there is no custom design available, agreed to follow existing music platforms as references
- compared layout, navigation, and content presentation approaches

**Search functionality:**
- discussed differences between search implementation on `Search` and `Playlist` pages
- reviewed possible filtering behavior for both pages
- discussed how search should interact with loaded data

**`Playlist` sorting:**
- discussed the possibility of adding sorting functionality on the `Playlist` page
- reviewed potential sorting options such as name, duration, and number of tracks
- discussed how sorting could improve usability

**SVG sprite preparation:**
- discussed preparing SVG icons for usage in the application sprite
- agreed to remove unnecessary metadata and unused attributes
- discussed removing redundant fills and other unused SVG content
- reviewed best practices for keeping SVG file lightweight and consistent

**HTTP and state management approach:**
- discussed working with Angular `HttpClient` and `Signal` Store
- reviewed Angular v22 resource-based APIs for data fetching
- discussed advantages and disadvantages of using resources
- agreed to focus on `HttpClient` and `Signal` Store to strengthen understanding of concepts required for Sprint 3 interview preparation
- discussed practical application of state management patterns in the project

**`CI/CD` and secrets management:**
- discussed moving `clientId` into GitHub Secrets
- reviewed required updates for `scripts/create-environment.sh`
- reviewed required updates for `.github/workflows/ci-cd.yml`
- discussed keeping sensitive configuration outside of the repository

**Team planning:**
- discussed the remaining work until the end of the sprint
- defined responsibilities for each team member
- agreed on task ownership and delivery expectations

**Team communication:**
- discussed rules for effective communication within the team
- reviewed expectations regarding task updates and code review communication
- discussed maintaining transparency about progress and issues

---

## What was decided

- To use a separate route for playlist creation instead of a modal-based approach
- To create a reusable horizontal card component
- To use Spotify and Deezer as UI references while adapting the design to project needs
- To implement dedicated search behavior for `Search` and `Playlist` pages
- To keep playlist sorting as a potential feature for improved usability
- To clean and optimize SVG files before adding them to the sprite system
- To distribute responsibilities across team members and follow the agreed sprint plan
- To use Angular `HttpClient` together with `Signal` Store for API interaction and state management
- To avoid Angular resource-based APIs at this stage in order to focus on concepts required for interview preparation
- To follow agreed communication practices and report blockers as early as possible
- To store `clientId` in GitHub Secrets and update deployment scripts and `CI/CD` configuration accordingly
