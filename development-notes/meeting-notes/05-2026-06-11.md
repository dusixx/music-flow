# Sprint 3 Planning Meeting: Forms, Angular v22 & Error Handling

**Date: 2026-06-09**

## What was discussed

**Sprint 3 requirements:**
- Forms
- Validation and displaying validation errors in templates
- Directives
- Pipes
- InjectionToken and custom providers
- Change detection with OnPush
- Discussed how these requirements fit into the current state of the project, since Firebase/Firestore had already been integrated and Angular v22 had just been released.

**Angular version upgrade:**
- discussed upgrading the project to Angular v22 shortly after its release
- noted that Angular v22 introduces new ways of working with forms using signals and resources
- discussed how the upgrade could affect the implementation of Sprint 3 and the preparation for Sprint 4
- discussed the fact that Firebase/Firestore integration already exists in the project

**Authentication UI implementation:**
- login and registration screens design
- creating reusable form components shared between authentication pages
- extracting shared authentication UI components and form logic

**Library implementation:**
- discussed playlist creation functionality
- looked at how playlist management works in Spotify and Deezer
- discussed possible approaches to playlist creation and editing

**API and services architecture:**
- discussed whether the current service structure would still work well as the application grows

**Error handling approach:**
- discussed whether services should handle and log errors internally
- questioned whether error logging inside services violates the single responsibility principle
- discussed whether errors should be handled where they can be shown to the user
- discussed different ways to show server-side errors:
  - directly inside forms/components
  - through global notifications (toasts)
  - via a centralized error handling mechanism
- discussed using Angular's ErrorHandler, HTTP interceptors, and HttpContext for future HTTP-based requests
- clarified that the current Firebase SDK calls return promises and are not based on HttpClient
- discussed catching Firebase errors in the components consuming these services
- noted that Angular resources include built-in error signals and agreed to look into them further
- agreed that Firebase error codes should be converted into user-friendly messages

**Form validation:**
- discussed Angular built-in validation capabilities
- considered whether third-party validation libraries should be introduced
- specifically mentioned Zod as a possible option
- discussed whether adding another library would be worth it at this stage of the project

**Repository management:**
- discussed introducing GitHub issue templates to standardize task creation and reporting

## What was decided

- To upgrade the project to Angular v22 and explore its new forms and resources APIs during Sprint 3 implementation
- To extract shared authentication UI components and form logic where it makes sense
- To implement playlist creation functionality inspired by Spotify and Deezer
- To continue refactoring API-related services as the application evolves
- Not to perform error logging directly inside Firebase-based services by default
- To handle Firebase SDK errors in components when user feedback is needed, while leaving room for a centralized solution in the future
- To display meaningful and user-friendly error messages instead of exposing raw Firebase error codes
- To postpone using additional validation libraries until Angular's built-in validation is evaluated
- To create GitHub issue templates for common tasks and workflows


## Open Questions

- Should server-side errors be displayed directly within forms or through a global notification system?
- Should the project use Taiga UI notifications or introduce an external toast library?
- How should Angular v22 resources and their built-in error signals be incorporated into the application's architecture?
- Is a third-party validation library such as Zod necessary, or will Angular's native validation be sufficient?
