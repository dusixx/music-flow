# Sprint-2 Planning Meeting: Architecture, Routing, Services & API

**Date: 2026-05-21**

## What was discussed

- Sprint-2 requirements:
  - Routing
  - Lazy loading
  - Route guards
  - Services (DI)
  - Signals

- Organization of services and API integration inside the already existing project structure

- Possibility of Jamendo API integration:
  - whether to integrate the API now
  - or temporarily use mock data until Sprint 4, where API integration becomes part of the technical requirements

- Implementation of a common `jamendo-api.service` responsible for base URL, client_id, common query params

- How to organize services inside the application:
  - which services should be global
  - which services should belong only to specific features/pages

- Possible implementation of `tracks.service`, `artists.service`, `albums.service` related to corresponding features/pages

- Reusability of services between features/pages

- Global services (`auth`, `player`) and possible abstract class implementation

- Service structure scalability to prevent chaos as the application grows

- Application deployment


## What was decided

- To create all main pages with minimal content in order to implement:
  - routing
  - lazy loading
  - route guards

- Not to store all services inside one common folder

- To use:
  - one shared service for API interaction
  - separate services inside related features

- To monitor how services grow and further separate the structure when necessary

- To integrate Jamendo API gradually if it doesn't slow down Sprint-2 implementation

- To set up application deployment on Netlify

- To configure CI/CD using GitHub Actions

- To review previously created backlog issues, create new tasks based on the decisions made during the meeting, and assign them to team members
