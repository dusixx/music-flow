# Sprint 2 Follow-up Discussion: Architecture, UI & Tech Decisions

**Date: 2026-05-22 - 2026-05-24**

## What was discussed

- Follow-up discussions after the mentor meeting about Sprint 2 implementation details
- UI and layout decisions:
  - decided to use a dashboard-style layout for desktop screens
  - this layout is not suitable for mobile devices
  - for mobile screens, a burger menu navigation will be used instead
  - discussed responsive layout behavior between desktop and mobile

- UI library change:
  - switched from Angular Material to Taiga UI
  - discussed using Taiga UI as the main UI kit for the project

- API approach (mocks vs real HTTP):
  - discussed whether to start working with real Angular HttpClient or use mock data
  - decided to use mock data for now
  - HTTP integration is planned for Sprint 4
  - agreed to follow the learning path represented by RS School and not introduce complexity too early
  - plan is to study Jamendo API documentation, inspect requests in Postman, and prepare for future integration

- Architecture discussions:
  - clarified separation between global and feature services
  - decided to build service structure step by step during development
  - agreed that full API structure can't be designed upfront

- API/service structure approach:
  - discussed 'one endpoint = one service' idea
  - services will be created gradually as features are developed
  - each service will belong to a specific feature when possible

## What was decided

- To use a dashboard layout on desktop and burger menu navigation on mobile
- To switch UI library from Angular Material to Taiga UI
- To work with mock data for now and postpone real HTTP integration to Sprint 4
- To keep evolving architecture during development instead of designing everything at the start
- To follow a gradual service approach:
  - one endpoint -> one service (when possible)
- To keep clear separation between:
  - global services (shared logic)
  - feature services (domain logic)
